# Cellpose(事前学習済み、ゼロショット)の導入検討 (2026-07-15〜)

## 背景

DWTの1epoch版を試している途中、「事前学習済みのCellposeをこのデータに直接(再学習なしで)
適用してみたらどうか」という話になった。Cellposeが実際に使っている手法自体は、
このプロジェクトが既に検討して「独立した情報にならない」と一度は退けた
「拡散ベースのflow場」([PROJECT_labeling3.md](PROJECT_labeling3.md)セクション7)に近い。
ただし、そこでの結論は「このプロジェクトの小さいデータ・自前実装で作った場合」の
話であり、**大量の多様な生物画像で事前学習済みの完成品を、再学習せずゼロショットで
試す**のは条件が全く違う、という考えで実際に試すことにした。

## Cellposeが何を学習しているか

Cellpose(Stringer et al., Nature Methods 2021、ここで使っているv4はCellpose-SAM/`cpsam_v2`)は、
2つのものを1つのネットワークで同時に学習する:

1. **前景確率**: そのピクセルが細胞か背景か(このプロジェクトの`binary`モードと同じ発想)
2. **中心方向のflow場(2ch)**: 各ピクセルから、そのピクセルが属する細胞の"中心"へ向かう方向ベクトル

## 「熱拡散シミュレーションで教師データを作る」とは

flow場の教師データ(正解)を作る手順が、単純な「距離変換の勾配」(このプロジェクトの
`_derive_unit_flow_from_depth`)とは違う。直感的に説明すると:

1. GTのインスタンスマスク(細胞1個分の塗りつぶし形)の中から、一番中心に近い点を1つ選ぶ
   (境界から最も遠い点)
2. その点に**熱源を置いて**、細胞の形の中だけで**熱を拡散させる**シミュレーションを行う
   (隣接ピクセルの温度を平均していく処理を何百回も繰り返す、と考えればよい。
   熱は細胞の輪郭の外には漏れ出さない)
3. 十分な回数拡散させると、中心が一番熱く、外側に向かって滑らかに温度が下がっていく
   "温度マップ"ができる
4. この温度マップの勾配(温度が上がる方向)を、そのピクセルの"正解"flow方向とする

**単純な距離変換の勾配との違い**: 距離変換(最短距離)は、細胞の形が凹んでいたりすると、
勾配の向きが形の凹凸に敏感に反応してギザギザになりやすい。熱拡散は物理的な拡散
現象なので、同じ凹んだ形でも勾配がより滑らかに(急に折れ曲がらずに)中心へ向かう
傾向がある。これが、単純な距離変換の微分と拡散ベースのflowが**数学的に別物**である理由。

**「循環しているのでは?」という疑問への回答**: 教師データを作る際にGTのインスタンス
マスクを使う(＝答えを使って教師データを作っている)のは事実だが、これは**学習時だけ**
の話。推論時にはGTマスクは要らず、学習済みモデルが予測したflow場だけを使う。
なので循環はしていない。

**このプロジェクトの結論との関係**: [PROJECT_labeling3.md](PROJECT_labeling3.md)セクション7で
「拡散ベースのflow場は独立した情報にならない」と結論づけたのは、あくまで
「GTマスクから作れる量は(前景,距離)の言い換えにしかならない」という数学的な話で、
**距離変換の勾配と熱拡散の勾配は別の量**なので、この結論がそのまま熱拡散にも
当てはまるとは限らない。むしろ今回の実測(下記)は、熱拡散ベースのflow場が
距離変換ベースのflowより実際に効いている可能性を示している。

## 推論時のインスタンス復元

各ピクセルの予測flow方向に沿って少しずつ動かす(軌跡積分)ことを何度も繰り返すと、
同じ細胞に属するピクセルは同じ収束点に集まる。収束点でグルーピングしてインスタンスにする。
これは、このプロジェクトが`distance_flow`モードで既に実装している`_integrate_and_cluster`
(main.py)と**全く同じアルゴリズム**。Cellposeの推論アルゴリズムそのものを、この
プロジェクトは既に一度再現していたことになる。

## 実装

`cellpose_track_and_visualize.py`に新規実装。既存の`track_and_visualize.py`の汎用部分
(Zスライス間タグ付け・面積ノイズフィルタ・描画)をそのまま再利用し、輪郭の作り方だけを
`cellpose.models.CellposeModel.eval()`に差し替えた(dwt_track_and_visualize.pyと同じ構成)。
このデータセットでの**再学習は一切行っていない**(事前学習済み重みをそのまま使用)。

```
python3 cellpose_track_and_visualize.py
```

モデルの重み(cpsam_v2、約1.15GB)は初回実行時に自動ダウンロードされる(以降はキャッシュ)。
1スライスあたりの推論は約1.5秒と高速(学習が要らないため、DWTのような数十分/エポックの
コストが発生しない)。

## 結果 (2026-07-15, Fluo-N3DH-CHO)

全10テストフレームに対しゼロショットで実行。全体としてはやや過剰予測気味
(例: t005でGT=9に対しpred=12)。

**重要な結果**: t005 z=0で、このプロジェクトが繰り返し失敗してきた既知の接触ペア
GT id=4/11が、**正しく2つの別インスタンスとして分離できていた**(タグ2↔GT11、
タグ4↔GT4が形・位置ともほぼ一致)。watershed・distance_flow・distance_flow_trained・
輝度ベース・DWT(1epoch版)を含め、これまで試した全手法がこのペアの分離に失敗して
いた中で、**再学習ゼロのCellposeが初めて成功した**。

## 「GT」表記についての重要な訂正 (2026-07-15)

上記の評価は、このデータセットで普段「GT」として表示・使用しているマスクに対する
ものだが、これは実際には**ST(Silver Truth、アルゴリズム生成の疑似正解)**だった
(main.pyのDATASET_CONFIGSで`Fluo-N3DH-CHO`は`annotation_subdir="ST"`)。
本物のGT/SEG(人手検証済み)はこのデータセットでは疎で、19個の(フレーム,Zスライス)
にしか存在しない。t005にはそもそも本物のGTが存在せず、GT4/11の分離が「本当に
正しいか」はSTを信じる限りでしか確認できない。結果自体は撤回しないが、
「STとの比較では」という限定付きで理解する必要がある。詳細は
[NOW.md](NOW.md)、result.htmlの表記修正はbuild_viewer.py参照。

## 本物のGT(19枚)での評価 (2026-07-16, `eval_real_gt.py`)

CTC(Cell Tracking Challenge)公式のSEG measure(各GTインスタンスに対し、
50%以上重なる予測領域があればJaccard指数、無ければ0点、の平均)で、本物の
GT/SEG19枚に対して評価した。

**公平性の注意**: 19枚中、train/val/testの分割でtestに入っているのはt033・t062の
2枚だけ。Cellposeはこのデータセットを一切学習していない(事前学習のみ)ので19枚
全部を公平に使えるが、自前で学習したモデル(DWT等)は残り17枚がtrain/valに
含まれている可能性がありリークになるため、公平な比較にはt033/t062の2枚だけを使う。

```
Cellpose(ゼロショット): 全19枚平均 SEG=0.785 ／ うちtest分割2枚のみ SEG=0.625
DWT(CHO, 1epochのみ):    test分割2枚のみ SEG=0.576(PROJECT_DWT.md参照)
```

同じ2枚・同じ指標での直接比較で、CellposeがDWT(1epoch)をわずかに上回った。
ただしDWTは1エポックしか学習していないため、本番学習後に逆転する可能性がある。
