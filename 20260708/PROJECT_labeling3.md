# 境界分離手法の再検討 (2026-07-14)

`PROJECT_labeling2.md`で、distance_transform(watershed)・distance_flow(後付け微分)・
distance_flow_trained(4ch学習済み方向)の3方式を試したが、いずれも接触細胞の
分離を安定して解決できなかった。ここでの議論を踏まえ、一度立ち止まって
問題を整理し直す。

## 1. watershedの構造的な弱点(ノイズ増幅とは別の、より根本的な問題)

これまで「モデルの予測ノイズが微分で増幅される」ことを過剰分割の原因として
説明してきたが、より正確には**watershed(および軌跡積分によるクラスタリングも
同様)は、そもそも「この2つの山(または収束点)が本当に別々の細胞なのか、
それともノイズで偶然2つに見えているだけの1つの細胞なのか」を判断する仕組みを
一切持っていない**、という構造的な欠陥がある。山(収束点)が2つあれば、
それが本物でもノイズでも無条件に境界線を引いてしまう。これが孤立した
GT id=1, 7, 15などが2〜3個に過剰分割された、より正確な原因の説明になる。

## 2. (未着手・未検討) 3次元的な一貫性を使う案

このデータはZスタック(3次元)なので、以下のような判断が理論上可能かもしれない
という着想が出た:

- 本物の接触(例: t005のGT id=4/11)は、複数の連続したZスライスにわたって
  同じような分割線が安定して現れる(実際にz=0〜4の5枚連続で確認済み)
- ノイズによる偶然の2山は、Zスライスが変わるとノイズの出方も変わるため、
  分割線がスライスごとに不安定・不連続になりやすいと考えられる

現在のパイプラインは、2次元1枚ごとに輪郭(インスタンス)をまず決定してから、
後付けでZ間をハンガリアン法で追跡する(`track_contours_across_z`)という順序に
なっており、**Z方向の情報が2次元の分割判断そのものには一切使われていない**。
これをどう使えば「本当に分けるべきか」の判断に活かせるかは、まだ具体案がなく
未検討。**この案は今は検討を進めない(2次元の問題を先に片付けてから)。**

## 3. (前景, 距離, dx, dy)という4チャンネル設計そのものへの疑問

distance_flow_trainedの結果が芳しくなかった一因として、設計そのものへの
疑問が挙がった:

**学習後のモデルが出す(前景, 距離, dx, dy)は、互いに数学的な整合性を
持つ保証がない。** GTの教師データを作る際は、dx, dyを「GTの深さから機械的に
導出した値」として作っている(`_derive_unit_flow_from_depth`)ので、GT同士は
定義上完全に整合している(dx, dy = ∇depth / |∇depth|が正確に成り立つ)。

しかし、モデルは4つのチャンネルを**それぞれ独立した損失(前景=BCE+Dice、
深さ・dx・dy=個別のMSE)で別々に学習している**。学習が(現実には決して
到達しない)完璧な収束に達すれば、4つの出力もGTと同様に整合するはずだが、
実際には各チャンネルにそれぞれ別々の残差誤差が残る。**「モデル自身が出す深さを
微分したもの」と「モデル自身が出すdx, dy」が一致する保証はどこにもない**
(それを強制する項が損失関数に無いため)。つまりモデルの出力4つを組み合わせて
使うこと自体が、そもそも整合性の怪しいデータ同士を組み合わせていることになる
可能性がある。

これが、distance_flow_trainedで過剰分割は直ったのに、肝心の分離効果まで
弱まってしまった一因かもしれない、という疑問が出ている。**結論は出ていない、
今後の検証課題。**

## 4. 次にやること(合意事項)

上記3の疑問を踏まえ、一旦4チャンネル設計から離れ、**(前景, 距離)の

watershed以外の2次元の境界の取り方を再検討する**、という方針になった。
3次元の話(セクション2)はいったん考えない。

## 5. 卓立度(prominence)によるフィルタの検討 → 否定的な結果 (2026-07-14)

watershedの種選び(`peak_local_max`)が、山の数だけで無条件に分割してしまう
(本物かノイズかを区別しない)という弱点に対し、地形の「卓立度」
(ある山から、それより高い山に登り直せるまでにどれだけ下る必要があるか。
`skimage.morphology.h_maxima`で測定)でフィルタする案を検討した。

**まず土台の確認**: 素のwatershed(`distance_transform`モード)でも、
t005 z=0で孤立しているGT id=1, 7, 12が実際に2つの輪郭に分割されていることを
確認した(過剰分割はdistance_flow系だけの問題ではなく、watershed自体が
既に抱えている問題だった)。

**閾値の検証可能性を先に確認**: 閾値を"提案"する前に、データが実際にどれだけ
あるかを数えた。このデータセット(train/val/test全92フレーム)で、**実際に
接触しているインスタンスの組は、独立した例としてはたった2組しかない**
(id 4/11がt000〜015の16フレームに渡って登場、id 18/26がt080〜087の8フレームに
渡って登場。これらは同じ2細胞ペアが時間を通して近くにいるだけなので、
本質的に2つの独立サンプルしかない)。

**1組目(id 4/11, t005 z=0)の実測**: 孤立細胞の余分な山は卓立度0.000〜0.100
(ほぼゼロ)、本物の接触ペアの山は0.480(id=4)・0.890(id=11)。ノイズと本物の間に
明確な"隙間"があるように見えた。

**2組目(id 18/26, t080 z=3)で検証(訂正: 最初の比較は誤りだった)**:

最初の確認で「id=18(0.670)が孤立細胞id=9の本来の山(0.680)と近い」ことを問題視し
「再現しなかった」と結論づけたが、これは**比較すべき対象を間違えていた**。
孤立細胞の"本来の"山がいくつであろうと、競合する2つ目の山が無ければ誤分割は
起きないため、比較対象にはならない。正しく比較すべきは「絶対に弾きたい
余分な山の最大値」と「絶対に拾いたい、接触ペアの本来の山の最小値」。

改めて全ピークを機械的に集計し直すと:

```
接触ペアの各メンバーの本来の山(最小) = 0.480 (1組目のid=4)
余分な山(孤立細胞のノイズ・接触ペア自身の余分な山も含めた最大) = 0.290 (2組目のid=5のノイズ)

→ 閾値を(0.290, 0.480]の間に置けば、2例とも正しく分離できる
```

**結論(訂正版)**: 2例に限れば、卓立度によるフィルタは実際に分離できていた
(「分離すらできない」という前回の結論は誤り、訂正する)。ただし、n=2で
たまたま矛盾なく収まる閾値が見つかった、というだけであり、3例目・4例目でも
同じ範囲に収まる保証は無い(否定されなかった、という弱い確認に留まる)。
接触例がこのデータセットに2組しかないという制約自体が、この種の閾値ベースの
手法を十分に検証すること自体を根本的に難しくしている、という限界は変わらず残る。

## 7. 曲率・面積・Cellpose式flow場の検討 → いずれも(前景,距離)から独立していないと判明 (2026-07-15)

(前景,距離)の2chから離れて、新しい独立した情報源を探す試みとして、以下を検討した。

- **曲率**: 輪郭(またはレベルセット)の曲率は、数学的には距離チャンネルの正規化した
  勾配場の発散(κ = ∇·(∇D/|∇D|))として書け、独立な情報ではなく(前景,距離)の
  微分にすぎないと判明した。
- **面積によるノイズ判定**: 卓立度や高さの代わりに、インスタンスの面積でノイズを
  判定する案。形に依存しない利点はあるが、Zスタックの最初・最後のスライスは
  本物の細胞でも必然的に面積が小さくなるため、本物とノイズを区別できない
  (2次元の1枚だけでは、Z方向の文脈がないと原理的に判断できない)。
- **Cellpose式の拡散ベースのflow場**: 各ピクセルから細胞中心へ向かうベクトル場を、
  GTのマスク内だけで熱拡散シミュレーションして作る案。しかし拡散には「どこまでが
  そのインスタンスの内部か」という情報(=インスタンス分割の答えそのもの)が必要で、
  これは推論時にはまだ使えない(循環になる)。学習データの作り方を工夫しても、
  結局(前景,距離)と同じGTマスクから作られる情報でしかなく、独立した学習
  チャンネルにする意味がない。

これらの検討から、「マスクの形状から作れる量は、どう工夫しても(前景,距離)の
言い換えにしかならない」という結論に至った。真に独立な情報源は、
(a)元画像の輝度・テクスチャデータ、(b)時間方向(他フレーム)のデータ、の
2つに絞られた。

## 8. 3次元化の検討: ボクセルの異方性 (2026-07-15)

このデータセット(Fluo-N3DH-CHO)の物理的なボクセルサイズは、TIFFのメタデータから
**x=y=0.202µm、z=1.0µm**と判明した(z方向はxy方向の約4.95倍)。3次元の距離変換を
行う場合、ボリューム自体をリサンプリングする必要はなく、
`scipy.ndimage.distance_transform_edt`の`sampling=(1.0, 0.202, 0.202)`引数を
使えば異方性にそのまま対応できる。

3次元化への期待(未検証): 現在のwatershedは「2つの山があれば無条件に分離する」
という構造的欠陥(セクション1参照)を持つが、3次元にすることで、ノイズによる
偽の山はZスライスをまたいで安定しない一方、本物の接触は複数のZスライスで安定して
現れる(セクション2で確認済み)という性質を、追加の工夫なしに活用できる可能性がある。
ただし保証はなく未検証。

議論の末、「まず2次元で決着をつけたい」という判断で、3次元化は一旦保留にした
(他学部生がcc3dのconnected_componentsで3次元に取り組む方針のため、
今回はそちらとは別の方向性を探る意図もある)。

## 9. 元画像の輝度データを使った輪郭分離「単純二値化(findContours) + 実データ」→ 分離失敗 (2026-07-15)

前景の判定はbinaryモデル(再学習なし)のまま、輪郭抽出だけを変更する新手法を
実装・検証した(`main.py`の`LABEL_MODE="binary_intensity"`、
`_binary_intensity_to_contour_input`)。モデルが前景/背景の判定にしか使っていない
輝度データを、輪郭分離の後処理段階で直接利用する狙い。

**手法**: 前景マスクの中で、元の入力画像の明るさをガウシアンぼかし(sigma=20、
細胞内部の模様によるノイズを均すため)した上で、watershedの地形として使う。
sigma=3では過剰分割(t005でGT9個→予測47個)がひどく、sigma=20まで強めてようやく
現実的な数になった。

**結果**: t005のフィルタ後合計数はbinary(10個)・distance_transform(16個)より
良い9個(GT=9と一致)になったが、**taggedパネルで実際に確認すると、GT id=4/11の
接触ペアは相変わらず1つの輪郭に統合されたままだった**。つまり合計数の改善は、
別の場所での分割具合が偶然帳尻を合わせていただけで、本来の目的(既知の接触ペアの
分離)には失敗した。

## 10. GTとの照合手段が存在しないという発見 (2026-07-15)

`track_and_visualize.py`の「③タグ付け手法: コスト(1-IoU)+ハンガリアン法」は、
GTとの照合ではなく、**予測同士(あるZスライスの輪郭と隣のZスライスの輪郭)を
照合してタグを一貫させるための処理**だった(`track_contours_across_z`、
`track_and_visualize.py:96-134`)。GTのマスクはこの処理に一切関わっていない。

`manifest.json`の`nGtTotal`/`nPredTotal`/`nPredFiltered`も、GTと予測をそれぞれ
独立に数えているだけで、相互の対応関係は一切見ていない。つまり、**特定のGT
インスタンス(例: id=4/11)が予測でも正しく分離できているかを直接検証する仕組みは、
現在のパイプラインのどこにも存在しない**。合計数の比較だけでは、compensating
errors(ある場所で余分に分割し、別の場所で分割不足になり、合計は偶然一致する)を
検出できない。セクション9の実験結果は、まさにこの限界が実際に表面化した例だった。
